# ch10 라이브러리와 모듈 (1)

## 10.1 라이브러리

> library: 프로그램 개발 시 활용할 수 있는 클래스와 인터페이스들을 모아놓은 것을 말한다.
> 

<aside>
📢

추가적인 설명 

**라이브러리(Library)와 API(Application Programming Interface)**

---

**라이브러리는 특정 기능이나 작업을 수행하기 위한 재사용 가능한 클래스/인터페이스의 집합이다.**

→ ***즉, 내 코드에 직접 포함시켜 작동시키는 실제 구현된 코드들의 묶음이라고 볼 수 있다.***

필요한 함수를 코드에 **직접** `import`**해서 호출하면서** 쓰고, 내 프로젝트에 포함되기 때문에 버전이 바뀌면 내 코드도 같이 수정될 수 있다.

```java
import pack1.A;
```

---

**API는 남이 만든 기능을 어떻게 호출하고 어떤 결과를 받는지 명시한 인터페이스이다.**

→ ***즉, 내부 구현 코드는 숨긴채 외부에서 사용할 코드만 열어둔 것이라고 볼 수 있다.***

보통 서버에 있는 외부 서비스(예: 날씨, 로그인, 결제 등)에 **요청을 보내고 응답을 받는 규칙**이고, 구현 내부는 몰라도 그 규칙대로만 호출하면 된다.

```json
방식(Method): POST

주소: http://127.0.0.1:8000/test-insert

데이터 형식:
{
	"device_id": "lidar_01",
	
	"x": 15.2,
	
	"y": -3.4,
	
	"intensity": 200,
	
	"stamp": 1710554732
}
```

</aside>

일반적으로 `JAR(Java ARchive)` 는 압축 파일`~.jar` 형태로 존재한다.

JAR 파일에는 클래스와 인터페이스의 바이트코드 파일`~.class`들이 압축되어 있다.

<aside>
✅

***JAR 파일***

자바 전용 ZIP 압축 파일이다.

*여러 개의 `.class` 파일, 리소스, 라이브러리 등을 하나의 압축 파일로 묶어 놓은 `.jar` 파일이다.*

**대표적인 용도**

자바 관련 파일들을 하나로 묶어서 자바 프로그램 및 애플리케이션을 쉽게 배포 및 실행하고, 다른 프로젝트에 라이브러리로 공유하기 위함이다.

</aside>

![image.png](ch10%20%EB%9D%BC%EC%9D%B4%EB%B8%8C%EB%9F%AC%EB%A6%AC%EC%99%80%20%EB%AA%A8%EB%93%88%20(1)/image.png)

특정 클래스와 인터페이스가 여러 응용프로그램에서 공통으로 사용되면, `JAR` 파일로 압축해서 라이브러리로 관리하는 것이 좋다.

인텔리제이에서는 Java Project를 생성하여 클래스와 인터페이스를 개발하고, `Artifacts`(아티팩트) 설정을 통해 최종 산출물인 `JAR` 파일을 만드는 기능이 있다.

- 프로그램 개발 시 라이브러리를 이용하려면 라이브러리 `JAR` 파일을 `ClassPath`에 추가해야 한다.
    - `ClassPath`: 클래스를 찾기 위한 경로

### Class path에 라이브러리 추가하기

- 콘솔(명령 프롬프트 또는 터미널)에서 프로그램을 실행할 경우
    - java 명령어를 실행할 때 `-classpath`로 제공
    - CLASSPATH 환경 변수에 경로를 추가
- 이클립스 프로젝트에서 실행할 경우
    - 프로젝트의 Build Path에 추가
- 인텔리제이 프로젝트에서 실행할 경우
    - Project Structure → Modules → Dependencies → 경로 추가

### 콘솔에서 -classpath 옵션 사용

> 맥OS
> 

classpath 구분자는 콜론이므로 콜론`:`과 함께 마침표`.`를 경로 뒤에 추가해야 한다.

```java
$ cd ~/ThisIsJavaSecondEdition/workspace/my_application_1/bin
$ java -classpath ~/ThisIsJavaSecondEdition/workspace/my_lib/dist/my_lib.jar:. app.Main
```

### 환경 변수 CLASSPATH 사용

- `-classpath` 옵션은 java 명령어를 실행할 때마다 별도로 추가해야 하는 불편함이 있다.
    - 그래서 아예 공통으로 사용하는 라이브러리에 경로를 추가해서 불편함을 없애야 한다.

> 맥OS
> 

사용자 홈 디렉토리에서 `ls -all` 명령어를 실행해서 `.bash_profile`을 찾아보고 없다면 생성한다.

(최신 맥은 `bash`가 아니라 `.zshrc` 권장)

```bash
$ touch .bash_profile
```

.bash_profile 파일을 텍스트 에디터`vi, nano`로 넣고 작성

```bash
export CLASSPATH=.:~/ThisIsJavaSecondEdition/workspace/my_lib/dist/my_lib_.jar
```

내용 적용

```bash
$ source ~/.bash_profile
```

이동 후 실행

```bash
$ cd ~/ThisIsJavaSecondEdition/workspace/my_application_1/bin
$ java app.Main
```

## 10.2 모듈

> Java 9부터 지원하는 패키지 관리 기능까지 포함된 라이브러리
> 

<aside>
📢

***패키지와의 차이점***

---

*패키지는 자바에서 클래스와 인터페이스를 논리적으로 묶어 놓은 디렉터리이다.*

즉 관련된 소스 코드 파일들을 묶어 분류하는 단위라고 볼 수 있다.

---

*모듈은 그보다 한 단계 위 개념으로, 외부에서 재사용할 수 있는 패키지들의 묶음이다.*

여러 패키지를 한 덩어리로 묶어서 공개범위와 어떤 외부 기능이 필요한지를 관리하는 더 큰 배포 단위라고 볼 수 있다.

***패키지 접근 제한자***

pack1.A를 public으로 했기 때문에 다른 패키지 pack2에서 사용할 수가 있어서 모듈이랑 다를게 없어 보일 수 있지만 이렇게 되면 외부에서도 사용할 수 있다. 

모듈을 이용하면 pack1.A가 public이더라도 외부 공개하지 않으면 외부에서 사용 불가능하다.

모듈에서의 공개 범위는 외부 프로그램에서의 접근 허용 여부를 결정하고, 패키지에서의 접근 제한은 해당 프로젝트 내부 클래스 간 접근 허용 범위를 결정한다.

</aside>

***일반 라이브러리와의 차이점***

라이브러리는 내부에 포함된 모든 패키지에 외부 프로그램에서의 접근이 가능하다.

모듈은 일부 패키지를 은닉하여 접근할 수 없게끔 할 수 있다.

![image.png](ch10%20%EB%9D%BC%EC%9D%B4%EB%B8%8C%EB%9F%AC%EB%A6%AC%EC%99%80%20%EB%AA%A8%EB%93%88%20(1)/image%2013.png)

- 모듈은 묘둘간의 의존 관계를 쉽게 파악할 수 있다.
    - 자신이 실행할 때 필요로 하는 의존 모듈을 모듈 기술자`module-info.java`에 기술

***의존 관계*** 

A 모듈은 B 모듈이 있어야 실행가능

B 모듈은 C 모듈이 있어야 실행할 수 있는 의존관계를 보여준다.

![image.png](ch10%20%EB%9D%BC%EC%9D%B4%EB%B8%8C%EB%9F%AC%EB%A6%AC%EC%99%80%20%EB%AA%A8%EB%93%88%20(1)/image%2014.png)

- 모듈도 라이브러리이므로 JAR 파일 형태로 배포할 수 있다.

![image.png](ch10%20%EB%9D%BC%EC%9D%B4%EB%B8%8C%EB%9F%AC%EB%A6%AC%EC%99%80%20%EB%AA%A8%EB%93%88%20(1)/image%2015.png)

- 대규모 응용 프로그램은 기능별로 모듈화(modulization)해서 개발할 수도 있다.
    
    → ***모듈화 방식으로 사용하면 재사용성 및 유지보수에 유리하다.***

## 10.4 모듈 배포용 JAR 파일

***모듈 개발을 완료했다면 다른 모듈에서 쉽게 사용할 수 있도록 `.java`가 아닌 바이트코드 파일`.class`로 구성된 배포용 JAR 파일을 생성해야 한다.***

→ 왜 `.class`로 구성해야 할까?

JVM이 즉시 실행할 수 있게 하고 원본 코드를 보호하기 위함이다.

## 10.8 리플렉션 허용

> 리플렉션: 컴파일러를 거치지 않고 실행 도중에 타입(클래스, 인터페이스 등)을 검사하고 구성 멤버를 조사하는 것을 말한다.
> 

*스프링 프레임워크에서 주로 사용*

어노테이션 처리, 객체 자동 생성 및 주입 등

- 은닉된 패키지는 기본적으로 다른 모듈에 의해 리플렉션을 허용하지 않는다.
    - 경우에 따라서는 은닉된 패키지도 리플렉션을 허용해야 할 때가 있다.
    
    ***exports와 opens의 차이***
    
    `exports`는 `import`해서 사용 `private` 접근 불가 
    
    `opens`는 `import` 불가능 접근 제어자 무시
    
    ### 허용 방법
    
    모듈 전체를 리플렉션 허용
    
    ```java
    open module 모듈명 
    {
    	…
    }
    ```
    
    지정된 패키지에 대해 리플렉션 허용
    
    ```java
    module 모듈명 
    {
    	…
    	opens 패키지1;
    	opens 패키지2;
    }
    ```
    
    지정된 패키지에 대해 특정 외부 모듈에서만 리플렉션 허용
    
    ```java
    module 모듈명 
    {
    	…
    	opens 패키지1 to 외부모듈명, 외부모듈명, …;
    	opens 패키지2 to 외부모듈명;
    }
    ```
    
- `export` 된 패키지는 언제든지 리플렉션이 가능하므로 `opens` 로 지정할 필요 없음
    - 은닉된 패키지 중에서 특정 패키지에 대한 리플렉션을 허용

## 10.9 자바 표준 모듈

자바 프로그램이라면 반드시 활용해야 되는 라이브러리 

→ ***JDK가 제공하는 표준 라이브러리***

![image.png](ch10%20%EB%9D%BC%EC%9D%B4%EB%B8%8C%EB%9F%AC%EB%A6%AC%EC%99%80%20%EB%AA%A8%EB%93%88%20(1)/image%2042.png)

https ://docs.oracle.com/en/java/javase/17/docs/api/java.se/module-summary.html

- `java.base`는 모든 모듈이 의존하는 기본 모듈이다.
    - `requires`하지 않아도 사용가능 다른 모듈은 모듈 기술자에 `requires` 명시하고 사용해야 함
    - `java.lang`, `java.util`, `java.io` 등의 핵심 패키지가 있으며, `java.lang`을 제외하고 `import`해서 사용 가능하다.
- `java.se`는 JDK가 제공하는 모든 모듈을 제공하는 집합 모듈이다.
    - Java 8 이전 버전과 같이 자바 표준 라이브러리를 제한 없이 사용하고 싶을 경우에는 `java.se`를 `requires`하면 된다.
    
    ```java
    module my_application 
    {
    	requires java.se;
    }
    ```
    

## 다른 방법

`thisisjava`프로젝트처럼 모듈 기술자가 없는 프로젝트를 만드는 방법이 있다.

→ 모듈 기술자가 없으면 모듈로 인식되지 않기 때문에 자바 표준 라이브러리를 제한 없이 사용할 수 있다.

### 표준 라이브러리를 모듈화한 이유?

---

- Java 8 이전 버전까지는 응용프로그램이 표준 라이브러리의 5%만 사용하는데도 불구하고 응용프로그램을 실행하려면 전체 표준 라이브러리가 갖추어진 자바 실행 환경`JRE`이 필요했다.

- 응용프로그램을 실행하는데 필요한 모듈만으로 구성된 작은 사이즈의 자바 실행 환경`JRE`을 만들기 위해서인데 작은 사이즈의 자바 실행 환경이 필요한 경우
    - 독립 실행형(응용프로그램 + 표준 라이브러리)으로 배포할 경우 표준 라이브러리의 크기가 작을수록 배포 사이즈가 줄어든다.
    - 제한된 자원만 가지고 있는 소형(임베디드) 기기에는 사이즈가 작은 자바 실행 환경이 필요하다.

![image.png](ch10%20%EB%9D%BC%EC%9D%B4%EB%B8%8C%EB%9F%AC%EB%A6%AC%EC%99%80%20%EB%AA%A8%EB%93%88%20(1)/image%2043.png)

자바 표준 모듈은 모듈 A에서 모듈 D까지 제공하지만, 프로젝트를 실행하는 데는 모듈 A와 모듈 B만 있으면 된다.

→ 따라서 모듈 C와 모듈 D를 제외하고는 프로젝트만 실행할 수 있는 작은 실행 환경을 jlink 명령어로 실행할 수 있다.

**추가적으로 Java 8의 콤팩트 프로파일(`compact1`, `compact2`, `compact3`)도 작은 사이즈의 자바 실행 환경을 위해 도입되었는데 `compact2`에 소속된 1개의 클래스만 필요한 경우에도 어쩔 수 없이 `compact1`보다 더 큰 `compact2`를 배포해야 하는 단점이 있다.**
